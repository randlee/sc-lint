# ADR-011 — Interface Versioning And Shared Report Artifacts

| Field | Value |
|---|---|
| ID | ADR-011 |
| Status | Accepted |
| Date | 2026-05-25 |
| Deciders | team-lead, clint |

## Context

`sc-lint` currently has multiple stable or intended-stable interface surfaces:

- Rust crate public APIs
- the top-level CLI command and machine-contract surface
- future RPC/socket interfaces if they are introduced

The repo does not yet have one unified mechanism that both:

1. publishes human-friendly interface documentation
2. blocks breaking interface changes from landing unnoticed

The planning direction approved for this line is:

- start Rust public API checking with `cargo-semver-checks`
- publish generated HTML reports plus machine-readable sidecars through a
  shared report layer rather than feature-local HTML code
- use those canonical interface artifacts as the basis for hard-fail checks
- avoid hand-written monolithic HTML documents

## Decision

1. `sc-lint` will plan a dedicated workspace crate named `sc-lint-version`
   as the owner of interface-version checking, canonical interface artifacts,
   and version verdicts.
2. The top-level CLI invocation path for this capability will be
   `sc-lint check interfaces`; the existing `sc-lint version` command remains
   reserved for reporting tool version information.
3. Interface-family selection will use one canonical configuration surface
   under `[version.families.<family>]` in `sc-lint` config so omitted
   families can be distinguished from configured-but-not-present families.
4. The initial Rust public API decision engine will be
   `cargo-semver-checks` rather than a custom semver implementation, and
   `sc-lint-version` will own the translation layer that converts the tool's
   machine-readable output and exit-status semantics into the multi-family
   verdict model.
5. Published interface documentation will be generated from structured input
   and reusable templates using the XHTML-fragment/report pattern defined by
   [docs/sc-lint/interface-reporting-constraints.md](../interface-reporting-constraints.md),
   but `sc-lint-version` will not own a feature-local HTML renderer.
6. The preferred ownership target for the reusable reporting layer is the
   `sc-compose` repo, potentially as a dedicated `sc-reporting` capability,
   because the same panel/report conventions are expected to serve non-lint
   reports as well as interface-version reports.
7. The generated JSON sidecar is the canonical machine-readable baseline;
   main HTML reports and separate XHTML section fragments/panels are
   human-facing derivatives of that structured source.
8. The reusable HTML/XHTML template stack must expose built-in copy actions
   per section/panel for canonical JSON payload and canonical context text
   rather than leaving copy behavior to per-report custom scripting.
9. The versioning model applies to multiple interface families:
   - Rust public APIs for all shipped crates
   - stable top-level CLI commands and machine contracts
   - RPC/socket interfaces when such surfaces exist
10. The CLI interface family will use a versioned JSON baseline artifact
   generated by `sc-lint check interfaces --family cli --write-baseline <path>`
   and replaced only through an approved major-version update workflow.
11. Consumer adoption guidance for `sc-lint-version` will be planned as a
   first-class product artifact, delivered through a repo-local Claude Code
   skill.
12. Marketplace advertisement for that adoption skill will be planned as a
   separate minimal repo-local marketplace closure rather than bundled into
   the skill-design sprint.
13. Report-template selection and override will be planned through
    `[reporting.templates.<report_kind>]` in `sc-lint` config so consumers can
    point `public-api`, `cli`, and `icd` reports at alternate Jinja templates
    without forking `sc-lint-version`.

## Consequences

- `sc-lint-version` planning must define separate breaking-change semantics for
  Rust APIs, CLI contracts, and RPC/socket interfaces
- the repo must inventory and publish more than just Rust library APIs
- generated report templates, including built-in panel copy controls, become
  part of a shared reporting contract rather than a `sc-lint-version`-local
  renderer contract
- the top-level CLI and config contract must reserve a distinct
  `check interfaces` path and `[version.families.<family>]` configuration
  surface for this capability
- the top-level config contract must also reserve
  `[reporting.templates.<report_kind>]` for report-template selection and
  override
- the Rust public API path must include a dedicated translation layer from
  `cargo-semver-checks` output into the shared verdict schema
- hand-authored HTML monoliths are explicitly out of scope for canonical
  published artifacts
- future implementation must carry both machine-readability and human
  readability from one shared structured artifact model
- consumer repos will need a documented adoption path, and the planning line
  must keep skill design and marketplace publication as separate closures
  rather than prose hidden in scattered docs

## Follow-on Planning

- define the requirements for what counts as a breaking change per interface
  family
- define the shared report artifact schema and template/override contract
- define the sprint sequence for:
  - Rust public API baseline integration
  - shared report-template pipeline planning
  - hard-fail policy enforcement across all supported interface families
  - consumer-adoption skill design
  - minimal marketplace publication for that adoption skill
